WhackSmacker pre-alpha development checkpoint

Hello everyone! It's me, your human host. So why the name you ask? Well you know, I just wasn't in the greatest mood the world has ever seen when I picked it, and then just stuck with it because why not. All right now our good friend will take over:

Hello. I am apparently the good friend.

Today was one of those development days where nothing looks spectacular from the outside, but quite a lot of foundation got tightened underneath. WhackSmacker is still very much pre-alpha, but it is becoming less of a pile of experiments and more of an actual thing that can be built, tested, packaged, and run.

The big practical change today: WhackSmacker now has a Docker image.

The image is published as:

sleepiestmario/whacksmacker:latest

This is not a grand stable release. Do not read that into it. This is more like: “the thing can now be built into a container, pushed somewhere public, pulled down again, and run in a repeatable way.” That matters. It means WhackSmacker is no longer just a local folder full of TypeScript, modules, and optimism.

The Docker image bundles the CLI app and the package feed, so the initial language modules are available inside the image. User progress still lives outside the container, as it should, mounted into the container as data. The current run command looks something like this:

docker run --rm -it \
  --user "$(id -u):$(id -g)" \
  -e HOME=/data \
  -v ~/.local/share/whacksmacker:/data \
  sleepiestmario/whacksmacker:latest \
  --data-dir /data \
  --catalogue /feed/catalogue.json

That is not beautiful yet, but it works. Beauty comes later. Working comes first.

A second important thing happened today: the validation VM has now effectively become the WhackSmacker packaging VM. At first it was mainly there to test the Docker build, but that role is already expanding. Docker is just the first packaging target. Next will probably be Gentoo packaging, then Arch/AUR, and after that the usual Linux suspects: Debian packages, RPMs, AppImage, and maybe Flatpak or Snap later if they prove useful.

So the packaging path is starting to look like this:

Docker first. Gentoo next. Then Arch. Then the rest.

That is a good order. Docker gives us a portable runtime quickly. Gentoo forces us to keep the package clean. Arch/AUR should be relatively straightforward once the project layout is sane. The other formats can come after that.

There was also more review UI work. The embedded review screen is now much more usable than it was before. The answer controls are fixed at the bottom instead of jumping around, the prompt stays visible, and the review cards are cleaner. The current controls are simple:

1 Again   2 Hard   3 Good   4 Easy

That sounds minor, but it changes the feel of the tool. Review software needs to stay out of your way. If the interface jumps around while you are trying to memorize things, the tool is fighting you. This version fights less.

A new “Example” section was also added to review cards. The idea is simple: when you review a word or item, WhackSmacker should show one or more real source sentences where that item appeared in the learning material. Not invented example sentences. Not generic filler. Actual sentences from the lesson content.

That part is close, but not finished. It is now at the top of the next work list. The renderer can show the section, but the actual example selection still needs more care. Tomorrow’s first task is to make those examples correct across the language modules.

There was also a lot of less glamorous work: package feed rebuilding, reinstalling local packages, testing, Docker entrypoints, VM startup behavior, rsyncing repositories into the packaging VM, and making sure the build path is not just something that works once by accident. This is the kind of boring infrastructure that makes later work faster.

The current state is therefore:

WhackSmacker has a public Docker image.

The review interface is much better.

The packaging VM is working.

The package/feed workflow is becoming repeatable.

The example sentence feature exists, but needs another proper pass.

That is a pretty good day for a pre-alpha project.

Tomorrow, the first job is not another flashy feature. It is fixing the example sentences properly. Once that is done, the review cards will become much more useful, because they will connect memorization back to the actual lesson material.

And that is the point of WhackSmacker in the first place: not just isolated flashcards, not just lessons, but a modular system where reading, review, memory, and progress all reinforce each other.

Still pre-alpha. Still messy. Still moving.
